# Data Model: Add Undo Functionality

## Undo Context

Represents the active working scope whose undo stack is currently selected.

**Fields**:

- `tabId`: Stable identity of the active tab or editor surface.
- `templateRef`: Existing template reference value with `name` and `version`,
  when applicable.
- `catalogRef`: Existing catalog reference value with `name` and `version`,
  when applicable.
- `themeRef`: Existing theme reference value with `name` and `version`, when
  applicable.
- `contextKey`: Derived stable key used to find the matching undo stack.

**Validation Rules**:

- The same visible tab and `*Ref` combination must produce the same context key
  for the duration of the app session.
- A changed tab, `templateRef`, `catalogRef`, or `themeRef` must produce a
  different context key when the visible editing context differs.
- Missing optional identities must be represented explicitly so unrelated
  contexts do not collide.
- Context keys must be derived from existing `*Ref` values instead of duplicate
  ID/version fields.

## Undoable User Action

Represents a completed user-originated action that changed application-visible
state and can be reversed.

**Fields**:

- `actionType`: Stable application action identity.
- `description`: User-facing summary of the change.
- `context`: Undo context active when the action completed.
- `changedTarget`: Stable identifier for the entity or state area changed.
- `diff`: Action-generated before/after diff needed by policy to undo or redo
  the state change.
- `completionStatus`: `completed`, `failed`, `canceled`, or `interrupted`.

**Validation Rules**:

- Only `completed` actions may become undo entries.
- The action must have a non-empty description and enough before/after diff data
  to restore or reapply state consistently.
- Payloads must use user input, stable identifiers, and plain data rather than
  framework, transport, filesystem, or component objects.

## Undo Entry

Represents one reversible completed action stored in a session undo stack.

**Fields**:

- `id`: Unique frame identity within the session.
- `contextKey`: Context key of the stack that owns the entry.
- `description`: User-facing summary of what undo will reverse.
- `actions`: One or more policy-owned reversal actions needed to undo the
  completed change.
- `diffs`: One or more action-generated before/after diffs for the completed
  mutation.
- `createdAtSessionOrder`: Monotonic session order used for diagnostics and
  deterministic tests.
- `persistenceStatus`: Whether the entry has been written to disk for the active
  session.

**Validation Rules**:

- Entries are appended only to the active context stack after successful action
  completion.
- Entries are reversed in last-in, first-out order within their context.
- Entries are written to disk as they change, may be released from memory after
  persistence, and are cleared on app startup rather than restored across
  sessions.

## Undo Diff

Represents focused before/after data generated by the action that performed the
mutation.

**Fields**:

- `actionType`: Action that generated the diff.
- `target`: Stable identifier for the mutated domain object or state area.
- `before`: Plain data needed to undo the mutation.
- `after`: Plain data needed to redo the mutation.

**Validation Rules**:

- Diffs are generated from the action that owns mutation context.
- Diffs must not store whole application state snapshots.
- Before and after data must be sufficient for policy operations to undo and
  redo without relying on stale component input.

## Undo Stack

Represents the ordered undo history for one context.

**Fields**:

- `contextKey`: Stable owner context.
- `entries`: Ordered collection of undo entries.
- `currentEntryId`: Current position in the ordered stack, or empty when the
  stack is before its first entry.
- `availability`: Whether undo or redo is currently possible for this stack.
- `storageLocation`: Active-session disk location or identifier for persisted
  stack information, when any entries have been written.

**Validation Rules**:

- A stack belongs to exactly one undo context.
- Undo, redo, and history navigation affect only the active stack.
- Inactive stacks remain unchanged during undo in another context.
- A returned context restores the same stack for the current app session.
- Adding a new entry after undo removes entries after the current position and
  preserves entries before the current position.
- Stack changes are written to disk as they occur.
- Prior-session persisted stack data is cleared on app startup.

## History Position

Represents the user's current location in an ordered stack.

**Fields**:

- `currentEntryId`: Entry currently applied, or empty when the state is before
  the first entry.
- `canUndo`: Whether an applied entry can be reversed.
- `canRedo`: Whether an undone entry can be reapplied.
- `nextUndoEntryId`: Entry that would be undone next, when present.
- `nextRedoEntryId`: Entry that would be redone next, when present.

**Validation Rules**:

- Redo reapplies undone entries in order.
- History-list selection moves by undoing or redoing intervening entries in
  order.
- Selecting a list item targets the state immediately after that item.

## Baseline Opened List Entry

Represents the synthetic first row in an active context's ordered history list.

**Fields**:

- `id`: Reserved constant `__undo-baseline__` (`UNDO_BASELINE_FRAME_ID`).
- `description`: `Opened <name>@<version>` from the active tab's primary ref, or
  `Opened` when no ref is available.

**Validation Rules**:

- Not stored as an undo frame; synthesized when building renderer summaries.
- Never persisted to disk and never pruned by stack push operations.
- Selecting it targets stack position before the first recorded entry
  (`currentIndex === -1`).
- Label is stored per active context in undo stack state (`currentBaselineLabel`)
  and refreshed on context switch.

## Undo Availability Summary

Represents read-only state displayed by menu, shortcut, status, or similar
renderer surfaces.

**Fields**:

- `activeContextKey`: Context currently summarized.
- `canUndo`: Whether the active context has an undoable entry.
- `canRedo`: Whether the active context has a redoable entry.
- `nextUndoDescription`: Description of the next action to undo, when present.
- `nextRedoDescription`: Description of the next action to redo, when present.
- `recentActions`: Ordered recent-action entries for the active context.
- `historyVersion`: Runtime version that changes when the active context stack
  changes.
- `baselineLabel`: Current baseline opened label for the active context.

**Validation Rules**:

- Summaries are derived from owned undo state and do not mutate history.
- Summaries update when the active context changes, an undoable action records,
  undo succeeds or fails, redo succeeds or fails, or history navigation
  succeeds or fails.
- Summaries must not expose entries from inactive contexts.

## History Transition Result

Represents the outcome of attempting to undo, redo, or navigate history.

**Fields**:

- `status`: `transitioned`, `not-available`, or `failed`.
- `mode`: `undo`, `redo`, or `go-to`.
- `contextKey`: Context where the transition was attempted.
- `entryId`: Entry targeted for reversal, reapplication, or navigation, when
  present.
- `message`: User-safe explanation when the transition is unavailable or fails.

**Validation Rules**:

- `transitioned` results may advance or rewind the stack position.
- `failed` results must preserve consistent application state and must not
  discard the entry unless the transition is confirmed.
- `not-available` results must not change application-visible state.

## Undo Workflow Participation

Represents one app workflow's explicit contract for creating undo entries.

**Fields**:

- `workflowArea`: App shell, catalog, template, theme, or common.
- `entryAction`: Named action that starts the user workflow.
- `policyOperation`: Operation that owns mutation and undo recording.
- `recordsUndo`: Whether the workflow creates an undo entry.
- `failurePaths`: Validation, cancellation, pending, and external-detail
  failures covered by tests or enforcement.

**Validation Rules**:

- Every completed state-changing user action in scope must declare undo
  participation.
- Non-state-changing actions must be classified as not undoable with rationale.
- Workflows that record undo must prove failure paths do not create incorrect
  entries.
