# Quickstart: Add Undo Functionality

## Prerequisites

- Read `specs/004-add-undo-functionality/spec.md`.
- Read `specs/004-add-undo-functionality/plan.md`.
- Review `specs/004-add-undo-functionality/contracts/undo-history-contract.md`.
- Review `specs/004-add-undo-functionality/contracts/undo-workflow-integration.md`.
- Treat existing undo files under `src/app/core/undo`,
  `src/domain/core`, `src/domain/operations/undo-operations`, and
  `src/domain/state/undo-stack` as migration evidence, not automatically
  compliant implementation.
- Use existing `TemplateReference`, `CatalogReference`, and `ThemeReference`
  shapes for content identity in undo contexts.

## Implementation Validation

Use tasks generated by `/speckit-tasks` to drive implementation. During the
work, validate each undoable workflow against these checks:

- user event enters through a named viewmodel callback
- non-trivial intent becomes a validated action
- handler case delegates to one focused controller use case
- controller orchestrates validations, state reads, and policy-owned operations
- policy operation records undo only after a completed reversible state change
- policy operation generates focused before/after diffs from the action context
- undo entries do not store whole application state snapshots
- stack changes are written to disk as they occur
- persisted stack data is cleared on app startup
- undo context uses existing template, catalog, and theme reference values
- inactive context stacks remain unchanged
- redo reapplies undone actions in order
- ordered history-list selection moves to the state immediately before the
  selected entry
- recording a new action after undo prunes previously undone entries while
  preserving earlier history
- renderer summaries remain read-only
- failed, canceled, interrupted, rejected, and pending actions do not create
  incorrect undo entries
- restart clears persisted prior-session undo history before any history is used

## Manual Acceptance Scenarios

1. In one `themeRef` context, perform at least three visible edits and invoke
   undo repeatedly.

   Expected result: edits reverse one at a time in last-in, first-out order.

2. Edit `themeRef` A, switch to `themeRef` B, perform another edit, and invoke
   undo.

   Expected result: only `themeRef` B changes. Returning to `themeRef` A
   restores its prior undo availability.

3. Edit one catalog context and one template context in different tabs, then
   switch tabs and invoke undo.

   Expected result: undo availability and reversal match the active tab and
   content context only.

4. Trigger validation failure, cancellation, pending duplicate action, and
   external-detail failure cases for a remediated workflow.

   Expected result: failed or incomplete actions do not appear in undo history.

5. Build undo history, close the application, reopen it, and inspect undo
   availability.

   Expected result: persisted prior-session undo state is cleared on startup and
   no prior-session undo entries are available.

6. Undo several actions, invoke redo, then complete a new undoable action.

   Expected result: redo reapplies undone entries before the new action, and the
   new action removes any remaining redoable entries while preserving earlier
   undo history.

7. Open the ordered recent-action list and select an older and newer item.

   Expected result: the app reaches the state immediately before the selected
   item by undoing or redoing intervening entries in order.

8. Build a large undo history, confirm stack data is written to disk, release
   eligible in-memory history data, then invoke undo, redo, and history-list
   navigation.

   Expected result: active-session history remains correct after data is loaded
   from disk-backed stack information.

9. Inspect recorded undo entries for representative workflows.

   Expected result: entries contain action-generated before/after diffs and do
   not contain whole application state snapshots.


Expected result:

- representative app shell, catalog, template, and theme workflows can undo
  and redo completed state-changing actions
- context switches restore the correct session stack using `*Ref` identities
- at least 20 consecutive actions can be undone in one context
- recent-action list selection navigates to the correct history position
- new actions after undo prune redoable branch entries
- failed or interrupted actions do not create incorrect undo entries
- undo stack changes are written to disk as they occur
- prior-session persisted history is cleared on startup and unavailable after
  restart
- undo entries use action-generated before/after diffs instead of whole
  application snapshots
- architecture and convention enforcement covers the new undo rules
- directives and Spec Kit artifacts have no known drift
