# Quickstart: UI Component Compliance Remediation

## Prerequisites

- Read `specs/003-ui-component-compliance/spec.md`.
- Read `specs/003-ui-component-compliance/plan.md`.
- Use `Todo.md` as the minimum authoritative unchecked component inventory.
- Preserve the completed queue and lifecycle decisions from
  `specs/002-constitution-compliance-remediation/plan.md`.
- When patterns conflict, prefer whole-app consistency using the constitution,
  completed `002` remediation, newer git-history evidence, and current compliant
  implementations.

## Planning Validation

1. Confirm the active feature pointer:

   ```powershell
   Get-Content .specify/feature.json
   ```

   Expected result: `feature_directory` is `specs/003-ui-component-compliance`.

2. Review the boundary contracts:

   ```powershell
   Get-Content specs/003-ui-component-compliance/contracts/component-workflow-boundaries.md
   Get-Content specs/003-ui-component-compliance/contracts/inventory-and-enforcement.md
   ```

   Expected result: the required component workflow flow, classification rules,
   evidence requirements, and enforcement triggers are clear before tasks are
   generated.

## Implementation Validation

Use tasks generated by `/speckit-tasks` to drive implementation. During the
work, validate each remediated workflow category against these checks:

- component event forwards to one named viewmodel callback
- non-trivial intent becomes a validated action
- handler case delegates to one focused controller use case
- controller orchestrates validations, read-only state access, and policy-owned
  operations
- business rules and mutation remain in policy-owned units
- external details stay behind replaceable seams
- local-only state is documented as local or moved to explicit coordinated
  ownership
- older working behavior is changed when it conflicts with the selected
  canonical pattern
- invalid input, stale input, repeated action, cancellation or close behavior,
  and external-detail failure are covered where applicable

## Final Validation

Before marking the feature complete:

```powershell
npm run lint
npm test
```

Expected result:

- all unchecked `Todo.md` component items have evidence-backed classification
- directly related inconsistent workflows have evidence-backed classification
- pattern conflicts record the selected canonical pattern and evidence
- lifecycle-triggered workflows follow the same named action flow as user
  interactions unless a documented exception exists
- passive renderer summaries remain read-only views over explicitly owned state
- representative common, shell, catalog, template, and theme workflows pass
- applicable failure paths are validated
- architecture and convention enforcement passes
- directives and Spec Kit artifacts have no known drift
